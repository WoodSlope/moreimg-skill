# 图文笔记内容

## 页面结构

### 封面页
- 页型：封面页
- 页面目标：保留用户原第 1 页结构，建立核心判断：AI 知识库建设不要一上来就只看 RAG。
- 入图文字：
  - 大标题：AI 知识库别急着上 RAG
  - 副标题：先理清内容源、切分方式、反馈闭环
  - 提示：企业知识库接入 AI 前，真正要先准备什么
  - 标语：知识库不是文件夹，RAG 也不是第一步
- 语义映射：用知识资产主线表达“先治理内容，再谈技术调用”。
- 避免误读：不要把 RAG 画成唯一主角；不要暗示只接入向量库就完成知识库建设。

### 内容页1/2
- 页型：流程页
- 页面目标：保留用户原第 2 页结构，把三层准备压缩成一条清晰的建设顺序。
- 入图文字：
  - 大标题：知识库建设的三层准备
  - 1 内容源：确认哪些资料能用
  - 2 结构化：决定怎么切分
  - 3 反馈闭环：让错误能被修正
  - 总结：不要把混乱搬进系统
- 语义映射：主关系是“内容源治理 -> 结构化切分 -> 反馈闭环”，表达准备顺序和递进关系。
- 避免误读：不要把三层画成并列能力清单；不要加入模型、向量库或 RAG 作为主步骤。

### 内容页2/2
- 页型：关系页
- 页面目标：保留用户原第 3 页结构，说明 RAG 是调用知识的技术路径，不是知识库建设起点。
- 入图文字：
  - 大标题：RAG 只是最后一步
  - 误解：接入 RAG = 有了企业知识库
  - 正确：RAG 之前还有内容治理、结构设计、反馈机制
  - 内容治理：决定什么能被用
  - 结构设计：决定怎么被检索
  - RAG：决定怎么被调用
  - 总结：先把内容变成可靠资产
- 语义映射：主视觉表现“内容治理”和“结构设计”作为前置基础，RAG 位于后段调用层，三者不是同等起点。
- 避免误读：不要把 RAG 放在中心支配全部；不要暗示内容治理和结构设计可以被 RAG 替代。

## 发布配文

很多企业做 AI 知识库，一上来就问：要不要上 RAG？用哪个向量库？

但真正影响效果的，往往不是技术组件，而是内容本身有没有被治理好。

更稳的顺序是：先确认内容源，再设计切分方式，最后建立反馈闭环。

RAG 是调用知识的技术路径，不是知识库建设的起点。
